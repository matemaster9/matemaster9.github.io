Java并发核心机制

volatile

定义

Java运行线程访问共享变量，为确保共享变量准确和一致的更新，线程需要排他锁单独获取这个变量。Java 提供了 volatile 来实现。当一个字段被声明为 volatile 时， Java 内存模型将会确保所有的线程看到的变量值都是一致的。

volatile 作为轻量级的 synchronized 修饰的变量具备以下特性：

1）线程可见性

2）禁止指令重排序

3）不保证操作原子性——所以 volatile 不能彻底保证线程安全

示例

public class Shutdown {

    public static void main(String[] args) throws InterruptedException {
        Runner one = new Runner();
        Thread thread = new Thread(one, "OneRunner");
        thread.start();
        TimeUnit.SECONDS.sleep(1);
        // 利用 volatile 标识位终止线程
        one.cancel();
    }

    static class Runner implements Runnable {
        private volatile boolean on  = true;
        private long i;
        @Override
        public void run() {
            while (on) {
                i++;
            }
            System.out.println(Thread.currentThread().getName() + " Count : " + i);
        }
        public void cancel() {
            on = false;
        }
    }
}

实现原理

被 volatile 修饰的变量，在进行写操作的时候，都会出现 lock 指令前缀！ lock 前缀的指令相当于一个内存屏障，在多核处理器下完成三件事情：

  • 它确保指令重排序时不会把其后面的指令排到内存屏障之前的位置，也不会把前面的指令排到内存屏障的后面；即在执行到内存屏障这句指令时，在它前面的操作已经全部完成；
  • 将当前处理器的缓存行内容写入内存
  • 上面这个写回内存的操作，会自动导致其他 CPU 的缓存的该内存地址失效。也就是对应的缓存行失效

synchronized

概述

synchronized 实现同步的基础，Java将每一个对象都当作一个🔒，具体为：

  • 普通同步方法，锁是当前实例对象
  • 静态同步方法，锁是当前类的Class对象
  • 同步块，锁是synchronized括号内配置的对象

实现原理

synchronized 在 JVM 的实现原理是：JVM 基于进入和退出 Monitor 对象来实现同步方法和同步代码块。

表现出synchronized针对同步块和同步方法两个进入方式：

  • 同步块实现使用了monitorenter和monitorexit指令
  • 同步方法实现使用了ACC_SYNCHRONIZED

注：==以上两种进入方式本质上都是对象监视器的获取==，对象监视器的获取是排他性的。

锁优化

synchronized 的四种锁状态

  • 偏向锁
  • 轻量级锁
  • 自旋锁
  • 重量级锁

偏向锁的目的：偏向于第一个获取锁的线程。当锁总是被同一个线程多次获取，为了让该线程获取锁的代价减小。

偏向锁的加锁过程：

当一个线程访问同步块时，JVM 会在对象头和栈帧的锁记录存储偏向的线程ID，此后该线程再次进入同步块时，就不在需要 CAS 进行加锁和解锁，只需要简单的测试以下 Mark Word 里是否存在指向当前线程的偏向锁！

在测试成功的情况下，表示当前线程已经获取锁

在测试失败的情况下，继续检查是否是偏向锁，在不是的情况下，才使用 CAS 竞争锁。如果已经是偏向锁了，则使用 CAS 将对象头的偏向锁指向当前线程。

偏向锁的撤销过程：

偏向锁采用的是：有竞争才释放锁的机制。撤销锁的时机：等待全局安全点（在这个时间点上不存在正在执行的字节码）

首先暂停拥有偏向锁的线程，然后检查持有偏向锁的线程是否处于running状态。

如果不处于，则将对象头设置为无锁状态

如果处于，拥有偏向锁的栈会被执行，遍历偏向对象的锁记录，栈中的锁记录和对象头的 Mark Word 要么重新偏向于其他线程，要么恢复到无锁状态或者标记对象不适合作为偏向锁

最后唤醒暂停的线程

轻量级锁加锁：

JVM 首先在当前线程的栈帧中创建锁记录空间，然后将对象头的 Mark Word 复制到锁记录中。

然后线程尝试将对象头的 Mark Word 替换为指向锁记录的指针（这里官方叫法：Displaced Mark Word）

如果替换成功，表示加锁成功。

如果替换失败，表示当前存在竞争， 当前线程可以尝试使用自旋获取锁

轻量级锁解锁：

解锁时，利用原子的 CAS 将Displaced Mark Word 替换到对象头。

如果成功，表示竞争没有发生

如果失败，表示当前存在竞争，锁会膨胀到重量级锁

自旋锁：

互斥同步进入阻塞状态的开销都很大，应该尽量避免。在许多应用中，共享数据的锁定状态只会持续很短的一段时间。自旋锁的思想是让一个线程在请求一个共享数据的锁时执行忙循环（自旋）一段时间，如果在这段时间内能获得锁，就可以避免进入阻塞状态。

自旋锁虽然能避免进入阻塞状态从而减少开销，但是它需要进行忙循环操作占用 CPU 时间，它只适用于共享数据的锁定状态很短的场景。

​ 表.锁的优缺对比

🔒 优点 缺点 适用场景
偏向锁 加锁解锁不需要额外消耗，几乎和执行非同步无差异 如果存在竞争、会导致锁撤销的销毁 只有一个线程访问同步块
轻量级锁 竞争的线程不会阻塞 始终竞争不到锁的线程会一致自选消耗CPU 追求响应速度、同步块的执行速度块
重量级锁 竞争线程直接阻塞，不消耗CPU 竞争线程阻塞、响应时间变慢 追求吞吐量

原子操作

循环 CAS 实现原子操作

package cn.matemaster.concurrency.second;

import java.util.ArrayList;
import java.util.List;
import java.util.concurrent.TimeUnit;
import java.util.concurrent.atomic.AtomicInteger;

/**
 * 基于循环 CAS 实现原子操作 线程安全的计数器
 * @author HUAWEI
 */
public class Counter {

    private final AtomicInteger atomicInteger = new AtomicInteger(0);
    private int i = 0;

    public static void main(String[] args) throws InterruptedException {
        final Counter cas = new Counter();
        List<Thread> list = new ArrayList<>(600);
        long start = System.currentTimeMillis();
        int gameOver = 100;
        for (int i = 0; i < gameOver; i++) {
            Thread t = new Thread(() -> {
                for (int j = 0; j < 10000; j++) {
                    cas.count();
                    cas.safeCount();
                }
            });
            list.add(t);
        }
        for (Thread thread : list) {
            thread.start();
        }
        TimeUnit.SECONDS.sleep(3);
        for (Thread thread : list) {
            thread.join();
        }
        System.out.println(cas.i);
        System.out.println(cas.atomicInteger);
        System.out.println(System.currentTimeMillis() - start);
    }


    private void safeCount() {
        while (true) {
            int i = atomicInteger.get();
            boolean suc = atomicInteger.compareAndSet(i, ++i);
            if (suc) {
                break;
            }
        }
    }

    private void count() {
        i++;
    }
}
